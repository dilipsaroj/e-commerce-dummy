Redux Toolkit was chosen over Context API because the assignment has two pieces of cross-cutting state with very different lifecycles: a cart that needs reducer-driven updates, selectors, and persistence, and a UI concern for the cart drawer. Context API could handle this, but it tends to blur state updates, derived totals, and persistence wiring once the cart grows beyond a trivial example. Redux Toolkit gave a better reviewer experience here: slice boundaries are explicit, actions are predictable, derived totals live in selectors, and the store shape stays intentionally small. That made it possible to satisfy the “do not store products in Redux” requirement cleanly while still keeping cart logic highly testable.

The biggest modeling decision was how to represent variants on top of the Fake Store API, which does not provide any variant structure. I used an enrichment layer that converts a raw API product into an `EnrichedProduct` with deterministic variants. Each variant carries its own id, color, size, stock count, and image set. This keeps the UI simple: the product detail page never has to invent data on the fly, and deep linking becomes a pure selection problem based on `color` and `size` in the URL. It also meant stock status could be derived consistently from the variant itself rather than from page-level state.

With additional time, I would focus on polish and production hardening rather than changing the architecture. I would add request caching and retry behavior, improve focus management and keyboard handling inside the cart drawer, and add a stronger visual test story around responsive layouts. I would also generate richer mocked galleries per variant instead of reusing the Fake Store image, and I would complete a deployment plus Lighthouse pass so the README could include real performance notes rather than implementation intent.
